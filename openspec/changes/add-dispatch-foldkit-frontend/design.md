# Dispatch FoldKit Frontend Design

## Current Evidence From Imports

The repo has three useful imported assets:

1. `imports/github/attune` is the nearest product ancestor. It is already FoldKit-native and includes `entry.ts`, `model.ts`, `message.ts`, `update.ts`, `view.ts`, route models, editorial page prose, and a clean editorial dark-mode visual language.
2. `imports/github/attune/repos/foldkit` provides the real package names: `foldkit@0.106.0` and `@foldkit/vite-plugin@0.8.0`. FoldKit depends on `effect@4.0.0-beta.78` and `@effect/platform-browser@4.0.0-beta.78`.
3. `imports/github/attune/repos/bulletproof-react` includes React markdown preview components. They are useful as a migration warning, not as an implementation template. Their shape is string markdown to sanitized HTML to `dangerouslySetInnerHTML`. Dispatch should instead parse/decode editorial source into a typed document model and render FoldKit nodes.
4. `imports/github/v0-web-page-mockup` is the canonical visual/page-grammar source for the current Attune product mockup. It is a Next.js 16 / React 19 / Tailwind / shadcn / Base UI app generated by v0. It has the page grammar we need to reimplement in FoldKit.

MDX is part of the Dispatch frontend architecture, but React is not. MDX is the agent/editorial DSL. Its JSX-like component tags are parsed into schema-decoded FoldKit component slots and rendered by FoldKit `Html`.

### Style Fidelity Rule

The v0 React mockup is the visual contract for the first FoldKit migration. The migration should preserve the existing style with minimal intentional changes:

- keep the clean editorial dark-mode palette
- keep the left sidebar composition
- keep the serif editorial headings and compact mono/code surfaces
- keep the page rhythm, gutters, and maximum widths
- keep `PageShell`, `PageHeader`, `Section`, `StatStrip`, `CodePanel`, `ListRow`, `FilterTabs`, `MetaGrid`, and `ActionBar` looking materially the same
- keep code panels and syntax coloring close to the mockup
- keep route-level information hierarchy for Discover, Workbench, Findings, Lineage, Exports, and Settings

Allowed changes are mostly inside page bodies where FoldKit MDX gives us a more consistent, agent-authorable component grammar. The shell and visual language should not be redesigned during this migration.

Tailwind utility strings are implementation evidence, not the new data model. They should be translated into stable FoldKit CSS classes and design tokens that preserve the visual result.

## FoldKit MDX Data Model

React and Next should not be the source of truth for Dispatch. The source of truth is FoldKit MDX compiled to a schema-backed page/document packet:

```text
DispatchMdxDocument
  sourcePath
  frontmatter
  blocks
  references
  componentRegistryVersion
```

The page layer adds:

```text
DispatchMdxPage
  route
  frontmatter
  blocks
  references
  layout
  componentRegistryVersion
```

The runtime does not evaluate arbitrary JavaScript. The accepted MDX subset is declarative:

```text
- markdown prose
- fenced code blocks
- static JSX-like component tags
- literal props
- arrays/objects only where schema allows them
- no arbitrary JS expressions
```

Blocks should be deliberately small but expressive enough to represent the v0 mockup:

- heading
- paragraph
- list
- callout
- code
- stat row
- stat strip
- filter tabs
- list
- code panel
- example pair
- action bar
- dispatch item reference
- issue reference
- artifact reference
- component slot

Inline nodes should be deliberately small:

- text
- emphasis
- strong
- code
- link
- issue link
- artifact link
- tag

The v0 React mockup is migration input. FoldKit MDX becomes the editable source form that agents can author and humans can review. The runtime compiler still fails closed through schemas before FoldKit sees anything.

The initial MDX component set should be intentionally close to the v0 primitives. A dumb agent should be able to author a useful page by composing familiar blocks instead of inventing layout:

```mdx
<PageShell width="lg">
  <PageHeader eyebrow="Discover" title="12 possible patterns in bulletproof-react" />
  <FilterTabs source="discover-status-tabs" />
  <Section label="Featured pattern">
    <PatternDossier patternRef="style-inline-primitives" />
  </Section>
  <Section label="All patterns">
    <PatternList source="current-run-patterns" />
  </Section>
</PageShell>
```

## Streaming MDX Materialization

Future dynamic pages should be generated as a component stream, not as a chat stream. The user should experience a page assembling into stable UI: shell first, header next, then cards, evidence panes, controls, code panels, and actions as their schema-decoded data arrives.

The stream unit should be a typed page patch:

```ts
type DispatchMdxPatch = Readonly<{
  readonly patchId: string
  readonly pageId: string
  readonly sequence: number
  readonly target:
    | { readonly _tag: "AppendBlock"; readonly afterBlockId?: string }
    | { readonly _tag: "ReplaceBlock"; readonly blockId: string }
    | { readonly _tag: "SetPending"; readonly slotId: string }
    | { readonly _tag: "RejectPatch"; readonly reason: string }
  readonly block?: DispatchMdxBlock
  readonly source:
    | { readonly _tag: "Agent"; readonly runId: string; readonly model?: string }
    | { readonly _tag: "System"; readonly runId: string }
}>
```

The important invariant is:

```text
model tokens
  -> component patch stream
  -> Effect Schema decode
  -> FoldKit model update
  -> materialized UI
```

Never:

```text
model tokens
  -> append raw assistant text to the page
```

This keeps generated pages agent-readable and human-readable. It also makes partial page states inspectable through FoldKit DevTools: agents can see which slots are pending, which patches decoded, which patches were rejected, and which component registry version produced the current screen.

Practical page loading behavior:

- Load the route shell immediately from the known registry.
- Materialize `PageHeader` and route-level navigation from the first valid patch.
- Use typed placeholder components such as `PendingSection`, `PendingEvidence`, and `PendingActionBar` rather than chat bubbles or skeleton prose.
- Replace pending slots only with schema-decoded components from the registry.
- Record rejected patches in model state for DevTools and later debugging, but do not render them as user-facing prose.
- Keep controls stable: buttons, tabs, and action bars should appear as controls, even if their backing data is still pending.

## FoldKit MDX Component Registry

MDX component tags should be constrained through a FoldKit registry:

```text
DispatchComponentName =
  | "PageShell"
  | "PageHeader"
  | "Section"
  | "StatStrip"
  | "CodePanel"
  | "CodeView"
  | "FilterTabs"
  | "ListRow"
  | "ActionBar"
  | "DispatchRiver"
  | "ThreadSummary"
  | "SafetyGate"
  | "FuzzerFinding"
  | "LinearIssue"
  | "AgentRun"
  | "OptimizationPacket"
```

Each component tag has schema-decoded props. Unknown component names fail decode. Unknown props fail decode. Raw JS expressions are not part of v0.

Example FoldKit MDX:

```mdx
---
route: workbench
title: Styling belongs in UI primitives and recipes
---

<PageHeader
  eyebrow="Workbench"
  title="Styling belongs in UI primitives and recipes"
  subtitle="Refine the examples and deterministic rule until this candidate is ready to promote."
/>

<StatStrip items={[
  { "label": "Matches", "value": "34", "icon": "scan-text" },
  { "label": "Reviewed", "value": "9", "icon": "eye" },
  { "label": "False positives", "value": "2", "icon": "x-circle" },
  { "label": "Runtime", "value": "180 ms", "icon": "timer" }
]} />

<Section label="Examples">
  <ExamplePair
    positiveRef="examples/style-card-positive"
    negativeRef="examples/style-avatar-negative"
  />
</Section>

<Section label="Deterministic rule" description="The ast-grep rule that encodes this pattern.">
  <CodePanel language="yaml" artifactRef="rules/no-inline-styling-in-ui-primitives" />
</Section>

<ActionBar
  note="YAML stays for inspection; intent drives revision."
  actions={[
    { "label": "View findings", "intent": "open-findings" },
    { "label": "Promote rule", "intent": "promote-rule", "tone": "primary" }
  ]}
/>
```

## FoldKit Rendering

FoldKit owns rendering and state transitions:

- `Model` contains selected route, filters, selected thread, and decoded fixture data.
- `Message` records user facts such as selected route, selected filter, selected thread, and acknowledged item.
- `view` is pure and renders `DispatchItem`, `WorkThread`, `DispatchMdxPage`, and `DispatchMdxDocument` as FoldKit `Html`.
- Runtime/data loading belongs outside `view`.

## React-to-FoldKit Migration Testing Contract

FoldKit's Story and Scene tests are part of the React-to-FoldKit migration plan. The React mockup should not be migrated by eyeballing alone, and each migrated route should carry its own FoldKit-native behavior and visible-contract tests.

Testing layers:

- schema tests for `DispatchMdxPage`, `DispatchMdxDocument`, component props, and rejected unsupported tags
- compiler tests for MDX source to decoded page data
- renderer tests for each v0-derived MDX primitive
- Story tests for route/page behavior, selected-route state, filters, and message transitions
- Scene tests for visible product information architecture and v0 style preservation

Scene tests should prove that the migrated pages preserve the visible contract:

- sidebar navigation renders the same route set
- Workbench shows page header, stat strip, example panes, deterministic rule pane, revise area, and action bar
- Discover shows filters, featured pattern, supporting examples, possible deterministic shape, known risk, and pattern list
- Findings shows review tabs, selected finding detail, deterministic selector, and review decision controls
- Dispatch shows event river, human action strip, safety states, and feed links

The tests should use FoldKit-native testing packages from the discovered FoldKit repo where possible, not React Testing Library. React components from `imports/github/v0-web-page-mockup` are source evidence only; they are not test dependencies.

## Mobile-First Surface

Default route is `/dispatch`. The first viewport should show the actual event river, not a landing page.

The phone layout:

- compact top rail with status counts
- segmented filters for all / review / safety / failed
- event river with severity, title, summary, references, and age
- persistent human-action strip when safety/review work exists

Desktop may add a thread/detail side region later, but mobile remains the baseline.

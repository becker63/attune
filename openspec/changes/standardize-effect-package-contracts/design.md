## Context

Attune currently has eleven active packages with a mix of generated shapes,
manual Effect services, package-local property tests, architecture policy
targets, Source BOM shards, and Joern-specific FastCheck/fuzz infrastructure.
The direction is clear, but the package boundary is still implicit: a package
can expose services, targets, generated files, tests, and environment-dependent
behavior without one canonical contract that Nx and policy gates can inspect.

The Attune architecture documents already define the loop:

```txt
Effect validates.
EventLog remembers.
Drizzle materializes.
Reactivity refreshes.
Atoms reason.
Nx grows the app.
```

This change makes that architecture feel like a framework. Effect DI defines
runtime services and dependencies. Effect Schema defines the authored boundary
and portable descriptor values. Reactivity and atoms define semantic package
view graphs. Nx defines deterministic generators, actions, affected execution,
and materialization. FastCheck applies adversarial evidence pressure to every
package boundary. SQLite/Drizzle stores local protocol facts as private
framework cache, not as a package authoring surface.

Attune Protocol should feel like a server framework or React-style programming
model. Package authors declare operations, schemas, services, views, laws, and
waivers. The framework derives obligations, hides runtime materialization, and
surfaces violations through TypeScript language-service diagnostics, quick info,
code actions, code lenses, and Nx checks. Users do not inspect React Fiber
during normal React authoring; Attune users should not inspect protocol
descriptor tables during normal Attune authoring.

The target flow is:

```txt
Package source files using framework DSL
  -> Nx framework materializer/generators
  -> Effect Schema descriptor values
  -> private Protocol Runtime
  -> local gitignored SQLite/Drizzle protocol cache
  -> ProtocolDiagnostics / ProtocolQuery projections
  -> framework projections:
       TypeScript language-service diagnostics, quick info, code actions, code lenses
       Nx CLI/check output
       optional future FoldKit UI
       optional future MCP adapter
       ephemeral debug/CI output only
```

This follows the Attune state philosophy:

```txt
Effect owns actions.
SQLite/Drizzle materializes protocol facts.
Reactivity announces freshness.
Atoms/views reason over materialized facts.
Language-service and Nx diagnostics project protocol deltas.
```

## Goals / Non-Goals

**Goals:**

- Define one canonical Attune package contract that every active package must
  expose.
- Define Attune Framework as a root `framework/` layer outside `packages/`.
  `framework/` defines the programming model; `packages/` consume it.
- Define Attune Protocol as the typed package/runtime boundary declaration and
  make package contracts the authored package-level boundary declarations that
  use the public framework DSL.
- Define private Protocol Runtime, Store, Obligations, Evidence, Delta,
  Diagnostics, and Query services behind framework APIs.
- Make the TypeScript language service the primary rich view over framework
  diagnostics, with Nx CLI/check output as the secondary view.
- Make the package contract the unit of uniformity, with operation metadata for
  public auditable boundaries rather than every private helper.
- Author package contracts as Effect Schema values, with Effect Schema as the
  source of truth for decoding, validation, and external ledger emission.
- Provide heavily typed contract builders that infer operation ids, Schema
  decoded/encoded types, valid law sets, view references, Effect RPC shapes,
  handler maps, evidence/replay types, and package test-layer requirements.
- Move every invariant that TypeScript can express into generated contract
  helpers, branded diagnostics, inferred law types, handler maps, and
  Effect-layer requirement types before adding architecture-policy checks.
- Prefer canonical `Effect.Service` service definitions generated by
  `@attune/nx`.
- Allow packages without real runtime services to expose minimal or empty
  package layers instead of manufacturing fake services.
- Allow lower-level `Context.Tag`/`Layer` definitions only through explicit
  package-contract waivers.
- Make Nx discover package contracts and derive targets, graph metadata, sync
  outputs, and policy inputs.
- Make every active package expose a package atom/Reactivity view graph for its
  meaningful read and reasoning state.
- Generate FastCheck harnesses from Effect Schema operation contracts.
- Generate Schema-coded internal package harnesses from package contracts so
  property and fuzz runners invoke packages through typed
  payload/success/error/observation/replay/evidence protocols. Runtime
  `@effect/rpc` remains an optional backend for that harness.
- Use Effect Schema's Arbitrary integration as the source of generated
  FastCheck inputs for package operations.
- Use `@fast-check/worker` for workerized property execution when package
  audits need parallelism, synchronous-loop protection, stronger isolation, or
  long-running fuzz pressure.
- Record property evidence for service operations, schema branches, operation
  laws, atom/Reactivity graph movement, state transitions, seeds, shrink paths,
  and counterexamples.
- Include the full Attune operation-kind taxonomy in the heavy migration,
  including policy, Joern, atom, event, generator, and resource-provider kinds.
- Migrate every active package to the canonical package contract shape.
- Reduce hand-maintained BOM complexity by moving final semantic truth into
  source declarations, generated source required by build/typecheck, and
  private local framework runtime/cache.
- Treat Source BOM and generator-shape manifests as legacy migration
  scaffolding or temporary compatibility views until the framework runtime and
  source-level generated artifacts replace them as workflow surfaces.
- Update `AGENTS.md` and every agent-facing workflow document so agents use the
  new framework-first, diagnostics-first, generator-first interface.

**Non-Goals:**

- Do not replace Effect DI with an Nx-owned service container.
- Do not make every service an Nx project; Nx remains package-granular unless a
  package is already split.
- Do not require every private helper, local pure function, test fixture, or
  implementation-only module to become a package-contract operation.
- Do not force pure data, schema, policy, or code-generation packages to invent
  runtime services only to satisfy DI shape.
- Do not ask agents to author complex conditional types by hand. The package
  contract helpers and generators own the heavy type inference.
- Do not make architecture policy duplicate invariants that typed contract
  helpers, Effect Schema decoding, Effect DI, or generated modules can reject
  directly.
- Do not require bespoke property tests before a generated service can be
  audited.
- Do not expose generated fuzz harness groups as stable product APIs. They are
  internal package harness protocols derived from package contracts; an RPC
  backend is optional.
- Do not make runtime `@effect/rpc` the root protocol primitive. It remains an
  optional generated audit harness backend until Effect 4 compatibility is
  resolved.
- Do not make FastCheck prove business correctness without declared laws,
  schemas, and operation metadata.
- Do not make raw V8/Istanbul percentages the primary correctness gate; they
  feed search and diagnostics while atom/Reactivity graph coverage remains the
  primary semantic contract.
- Do not build a native binary AFL/libFuzzer clone; coverage-guided search
  remains TypeScript/Nx/FastCheck/Effect evidence infrastructure.
- Do not replace Effect Schema arbitraries with a separate generator catalog.
  Workerized property execution still consumes arbitraries derived from Effect
  Schema.
- Do not require every commit-tier property to use per-predicate workers.
  Worker isolation is selected by target tier and package risk.
- Do not move durable writes, resource execution, or scheduler behavior into
  atoms. Atoms remain read/reasoning surfaces over Effect-owned writes and
  Reactivity-owned freshness.
- Do not require checked-in ProtocolDelta reports, obligation reports, evidence
  summaries, architecture summaries, or Markdown/JSON status reports as the
  workflow. Debug and CI reports may be emitted ephemerally only.
- Do not make MCP a core framework surface. It may become a future optional
  adapter over framework diagnostics/query services.
- Do not remove Joern-gated fuzz campaigns; they become first-class
  proof-pressure/fuzz targets behind the same evidence model.
- Do not require all property targets to run on every quick local edit. The
  contract defines tiers for commit, push, proof-pressure, and nightly use.

## Decisions

### Decision: Attune Protocol is a framework, not an integration platform

Attune Protocol should feel like a server framework or React-style programming
model. Package authors declare operations, schemas, services, views, laws, and
waivers. The framework derives obligations, hides runtime materialization, and
surfaces violations through TypeScript language-service diagnostics, quick info,
code actions, code lenses, and Nx checks.

The framework internals may maintain descriptor hashes, obligation graphs,
generated artifact state, evidence events, and deltas in a local SQLite store.
Those internals are not normal package authoring surfaces. Package authors and
agents repair framework diagnostics, not raw protocol rows or checked-in
reports.

The Attune Framework vocabulary is:

```txt
Attune Framework
  The root framework/ layer that defines the programming model for package
  protocols, runtime materialization, local protocol store, diagnostics, Nx
  materialization, and testing/evidence support.

Attune Protocol
  The authored boundary declaration exposed by packages through
  src/attune.package.ts, using the framework DSL.

Protocol Runtime
  Private framework runtime that materializes descriptors, obligations,
  evidence, generated artifact state, waivers, and deltas into a local SQLite
  store.

Protocol Diagnostics
  The primary public projection over Protocol Runtime state: language-service
  diagnostics, quick info, code actions, code lenses, and Nx check output.

ProtocolDelta
  Internal/runtime fact model and diagnostic source. Package authors and agents
  may see it through diagnostics, quick info, or debug commands, but they must
  not hand-edit or rely on checked-in delta reports.

Framework Views
  Primary: TypeScript language service.
  Secondary: Nx CLI/check output.
  Future optional: FoldKit UI, MCP adapter, ephemeral debug reports.
```

### Decision: Protocol IDs are mostly deduced from typed source declarations

The authored protocol surface should prefer symbols, object references, and
typed declarations over manually threaded string references. Stable string IDs
still exist, but they are serialized identities, not the primary authoring
mechanism.

Authoring should prefer:

```ts
export const projectionChanged = reactivityKey("discovery.projection")

export const discoveryReadModel = baseAtom({
  refreshesOn: [projectionChanged],
  output: DiscoveryReadModel,
})

export const workbenchSnapshot = packageViewAtom({
  reads: [discoveryReadModel],
  output: WorkbenchSnapshot,
})

export const eventReplayProjection = projection({
  input: DiscoveryEventSequence,
  output: WorkbenchSnapshot,
  writes: [projectionChanged],
  laws: inferLaws(),
})
```

Over:

```ts
projection("event-replay-projection", {
  touches: ["workbenchSnapshotAtom"],
  laws: ["projection.deterministic-replay"],
})
```

The framework materializer computes:

```txt
operation id
operation kind
stable descriptor identity
input/output/error types
inferred laws
obligations
reachable view graph
handler/property registry entries
type-guidance partitions
generated artifact ownership
evidence expectations
diagnostic source ranges
repair actions
```

Explicit stable ID overrides remain available when history/cache/replay
identity must survive source renames.

### Decision: Strings survive at serialization boundaries, not authoring edges

Keep stable IDs for:

```txt
cache keys
descriptor hashes
evidence/replay identity
diagnostic codes
Nx target names
artifact manifests
external process/file/worker boundaries
debug output
```

Do not require humans to manually cross-reference those strings inside package
declarations when a symbol reference can express the same relationship.

Examples:

```txt
Operation authoring:
  prefer eventReplayProjection symbol
  serialize "event-replay-projection"

View authoring:
  prefer workbenchSnapshot atom symbol
  serialize "workbenchSnapshotAtom"

Service authoring:
  prefer DiscoveryEvents service value
  serialize "attuned-discovery/DiscoveryEvents"

Law authoring:
  prefer inferLaws() or typed law descriptor
  serialize "projection.deterministic-replay"

Waiver authoring:
  prefer waive(CocoIndexClient, waivers.contextServiceShape(...))
  serialize category and target ID
```

### Decision: Static framework declarations define intent; implementation analysis verifies

The materializer may parse source files, resolve symbols, inspect imports, and
derive source ranges, but it must not treat arbitrary implementation scanning
as the semantic source of truth.

The rule is:

```txt
Declarations define intended semantics.
Static analysis verifies implementation shape.
Runtime evidence proves behavior.
```

Allowed derivation:

```txt
reactivityKey(...) declarations
baseAtom(...) declarations
derivedAtom(...) declarations
packageViewAtom(...) declarations
projection(...) declarations
resourceProvider(...) declarations
generator(...) declarations
policyRule(...) declarations
Effect service references
Effect Schema references
typed waiver constructors
```

Risky inference that should remain validation/suggestion only:

```txt
guessing operation writes by scanning arbitrary function bodies
guessing package-level atoms by naming convention only
guessing destructive provider semantics from command strings
guessing laws from implementation branches without contract metadata
```

The framework should use the TypeScript compiler API, language-service APIs, or
ts-morph-style extraction to resolve symbols, imports, source ranges, and type
information where needed. Declarations remain the source of semantic intent.

### Decision: View edges are derived from the declared dataflow graph

Package authors should declare semantic roots and view graph nodes. They should
not manually maintain every downstream atom/view reference.

Example:

```ts
const projectionChanged = reactivityKey("discovery.projection")

const readModel = baseAtom({
  refreshesOn: [projectionChanged],
  output: DiscoveryReadModel,
})

const workbenchSnapshot = derivedAtom({
  reads: [readModel],
  output: WorkbenchSnapshot,
})

const replay = projection({
  input: DiscoveryEventSequence,
  output: WorkbenchSnapshot,
  writes: [projectionChanged],
})
```

The framework derives:

```txt
replay
  writes projectionChanged
  refreshes readModel
  recomputes workbenchSnapshot
```

This derived edge creates view-movement obligations and diagnostics.

### Decision: The protocol reference ladder

```txt
Level 0:
  raw string IDs everywhere

Level 1:
  branded string IDs

Level 2:
  literal-preserving builders

Level 3:
  symbol/object references

Level 4:
  static DSL extraction through TypeScript compiler/language-service APIs

Level 5:
  generated exact registries over the operation tuple

Level 6:
  Effect Schema-coded runtime boundaries

Level 7:
  generated invocation harness or optional RPC backend

Level 8:
  language-service diagnostics from obligations minus evidence
```

The framework target is Level 3-8. Level 0-2 may exist only as migration
scaffolding or serialized artifacts.

`packages/`. `framework/` defines the programming model. `packages/` consume the
programming model.

### Decision: Framework internals are hidden behind public DSL and diagnostics

Product packages import the framework DSL and generated local artifacts. They
do not import SQLite tables, Drizzle clients, ProtocolStore internals,
materialization internals, language-service internals, or Nx internals. The
runtime/store/diagnostic implementation is private framework machinery.

Normal package authors should see:

```txt
defineAttunePackage
operation builders
view builders
law helpers
evidence producer helpers
Effect service/layer helpers
language-service diagnostics
code actions
Nx check output
```

Normal package authors should not see descriptor hashes, obligation derivation
internals, generated artifact manifests, evidence indexes, diagnostic
projection internals, protocol store tables, internal Source BOM history, or
generator-shape history.

The import boundaries are:

```txt
Allowed:
  packages/* -> @attune/framework-protocol
  packages/* -> generated local framework artifacts
  packages/* -> @attune/framework-testing only for package evidence tests

Not allowed:
  packages/* -> @attune/framework-sqlite
  packages/* -> @attune/framework-runtime/internal
  packages/* -> @attune/framework-nx/internal
  packages/* -> @attune/framework-language-service
  packages/* -> raw Drizzle tables
  packages/* -> ProtocolStore internals

Allowed:
  framework/language-service -> framework/runtime query/diagnostic services
  framework/nx -> framework/protocol + framework/runtime
  framework/runtime -> framework/protocol + framework/sqlite
  framework/testing -> framework/protocol + framework/runtime test APIs
  framework/sqlite -> framework/protocol schema values
```

Public framework authoring should look like:

```ts
import {
  atomFamily,
  command,
  defineAttunePackage,
  eventFacade,
  generator,
  inferLaws,
  policyRule,
  projection,
  query,
  resourceProvider,
  views,
} from "@attune/framework-protocol"
```

These imports are not allowed from product packages:

```ts
import { protocolDeltasTable } from "@attune/framework-sqlite/internal"
import { computeObligationGraph } from "@attune/framework-runtime/internal"
import { drizzle } from "@attune/framework-sqlite"
```

Drizzle is never a public app/package API. Only framework internals and
Nx/language-service integration depend on runtime/sqlite internals.

### Decision: Language service is the primary rich framework view

The TypeScript language service is the primary rich view over the framework
runtime. It turns protocol deltas and evidence state into file-positioned
diagnostics, quick info, code actions, and code lenses. It calls framework
query services and does not reimplement protocol semantics. Code actions call
deterministic Nx generators/checks and never hand-edit generated artifacts.

The authoring loop is:

```txt
1. Package author edits src/attune.package.ts or generated framework-facing code.
2. Framework materialization updates local protocol runtime/cache.
3. Language service displays diagnostics at the authored boundary.
4. User/agent applies code actions that invoke deterministic Nx generators.
5. Nx checks run.
6. FastCheck/evidence producers record evidence.
7. Diagnostics disappear or narrow.
```

The framework exposes errors as framework diagnostics, not raw runtime rows.
Example:

```txt
attune/protocol/missing-property-evidence

Operation "event-replay-projection" declares projection semantics, but no
evidence producer observes deterministic replay or workbenchSnapshot movement.

Why:
  operation kind = projection
  inferred law = projection.deterministic-replay
  touched view = workbenchSnapshotAtom

Suggested action:
  Generate protocol property evidence scaffold

Validation:
  nx run attuned-discovery:check
```

Expected language-service capabilities:

```txt
diagnostics:
  missing operation metadata
  stale generated source
  missing evidence producer
  missing atom graph edge
  dead Reactivity key
  invalid law
  hidden command/env boundary
  weak oracle
  high-rejection filter
  incomplete type guidance

quick info:
  operation obligations
  inferred laws
  touched views
  evidence status
  package layer requirements
  generated files owned by this declaration

code actions:
  generate operation registry
  generate atom view edge
  generate property evidence scaffold
  add missing type-guidance partition
  add waiver skeleton
  refresh protocol materialization

code lenses:
  "3 missing obligations"
  "evidence: 4/7 obligations observed"
  "stale generated source"
  "open generated operation registry"
```

The language service must not require MCP, checked-in ProtocolDelta files, or
checked-in report files.

### Decision: Protocol reports are ephemeral, not checked-in workflow artifacts

The framework does not require checked-in ProtocolDelta reports, obligation
reports, evidence summaries, architecture summaries, or Markdown/JSON status
reports. The authoritative protocol state lives in source declarations,
generated source required by the build, and the local gitignored protocol
runtime/cache.

Recommended runtime/cache locations are:

```txt
.attune/cache/protocol.sqlite
.attune/cache/protocol/*
.nx/cache/*
```

Diagnostics are projected through the language service and Nx output. CI may
upload ephemeral artifacts and debug commands may print or write temporary
reports under gitignored cache paths, but reports are not source truth and are
not committed.

The implementation substrate remains Effect-native:

```txt
Effect.Service / Context / Layer / R
  protocol implementation and requirement tracking

Effect Schema
  protocol descriptor, evidence, replay, diagnostic, and tool codec

Effect Schema Arbitrary
  generated property inputs

Effect Schema JSON Schema / annotations / equivalence
  external descriptors, docs, equality/change detection, type-guidance

@effect/platform
  typed filesystem/command/path/terminal/config boundaries

@effect/sql or Drizzle + Effect services
  local protocol store and later production Postgres seam

@effect/rpc
  optional/future generated audit harness backend, not the root primitive

@effect/vitest
  layer-aware protocol conformance tests

@effect/language-service
  editor/agent diagnostics as the primary rich framework view
```

### Decision: Local SQLite protocol store is the private compiler database

The Attune Protocol Store is the local SQLite/Drizzle-backed persistence layer
owned by framework internals. It stores descriptors, obligations, generated
artifact records, evidence runs/events, deltas, waiver state, and repair
suggestions. It is the local compiler/runtime database for architecture
obligations and evidence, not the production product database and not a product
package API.

SQLite is the first backend because it is deterministic, cheap, inspectable,
and suitable for local agent/compiler state. A later PostgreSQL layer may
satisfy the same service API, but Postgres is not a blocker for the local
protocol runtime.

Drizzle is hidden behind Effect services. Ordinary packages, agents,
language-service views, and Nx checks depend on high-level framework services
such as `ProtocolRuntime`, `ProtocolQuery`, `ProtocolProjection`, and
`ProtocolDiagnostics`, not raw Drizzle. Generated/internal modules may use a
low-level Drizzle client only through the framework/sqlite boundary.

Rows that cross runtime, file, process, or agent boundaries are Schema-coded:
protocol descriptors, obligations, generated artifact records, evidence events,
deltas, waivers, repair actions, tool inputs/outputs, diagnostics, and replay
metadata.

### Decision: The package contract is the unit of uniformity

The completed migration standardizes package boundaries, not every internal
symbol. A package contract is the Schema-coded package-level descriptor and
materialized contract for an Attune Protocol boundary. It inventories public
auditable operations: exported service operations, generator entrypoints,
resource/provider actions, policy rules, codecs, projections, atom-family
boundaries, query/command boundaries, and other functions that cross package
boundaries or mutate/observe meaningful package state.

Private helpers, local pure functions, implementation-only classes, and fixture
plumbing stay ordinary TypeScript. They are covered indirectly when generated
property harnesses exercise the public auditable operations that call them, and
V8/Istanbul feedback can still report dead or unexplored implementation paths.

Rationale: the contract should remove ambiguity for agents without turning the
repo into a hand-maintained metadata catalog. Package boundaries are where
runtime dependency, generated provenance, semantic view movement, and external
tooling behavior become architectural facts.

Alternative considered: require every exported function and helper to appear in
the contract. Rejected because it creates metadata churn, encourages fake
operation entries, and makes agents spend attention on implementation detail.

### Decision: Every active package exports an Attune package contract

Each package will expose a generated or sync-maintained contract module at:

```txt
packages/<name>/src/attune.package.ts
```

The contract describes the package boundary:

```txt
package id
source root
package kind
public entrypoints
services
operations
Effect layers
schemas
operation-kind metadata
inferred law descriptors
custom law extensions
semantic coverage expectations
Reactivity keys
base atoms
derived atoms
package view atoms
atom graph coverage expectations
generator provenance
waivers
```

The module exports at least:

```txt
PackageContractSchema
PackageContract
PackageLayer
PackageTestLayer
```

`PackageLayer` composes package-owned services and leaves external requirements
visible to Effect DI and graph derivation. `PackageTestLayer` composes a fully
provided deterministic test boundary suitable for generated property audits.
Leaf/application packages may also expose `PackageRuntimeLayer` when a fully
provided runnable layer is useful, but that layer is optional and must not hide
the dependency facts used for graph derivation.

Packages that are pure schema, policy, generator-template, fixture, or data
packages may expose a minimal package contract with `PackageLayer = Layer.empty`
or another generated no-op layer when they do not own real runtime services.
Those packages still declare public auditable operations, schemas, generated
provenance, package view atoms when meaningful, and property evidence
expectations.

Rationale: agents need one place to discover the package shape. A generated
contract lets policy check the boundary without manually parsing every local
idiom.

Alternative considered: keep package metadata only in Source BOM shards.
Rejected because the BOM is useful evidence, but it should not become a second
source of truth for runtime services and layers.

Alternative considered: infer everything from raw TypeScript AST. Rejected
because AST discovery is useful for drift checks, but the package must declare
semantic intent such as operation kind, laws, and coverage labels.

### Decision: Every package exposes an atom/Reactivity package view graph

Every active package exposes a generated or sync-maintained package view graph
through its contract:

```txt
operations
  -> events/evidence/projection writes
  -> Reactivity keys
  -> base atoms
  -> derived atoms
  -> package view atoms
```

This graph is the primary semantic observation surface. The shape does not mean
every package has UI state. It means every package exposes its meaningful
read/reasoning state as atoms and connects writes to freshness through
Reactivity.

The graph is package-level. It names semantic package views that humans and
agents should reason about, not every internal read model or temporary local
value. A package can satisfy the contract with a small set of stable view atoms,
such as package status, evidence, coverage, readiness, registry, or domain view
atoms, as long as meaningful public operation effects are observable through the
declared graph.

Examples:

```txt
attune-nx
  generator plan atom
  generated file diff atom
  generator provenance atom

attune-architecture
  policy findings atom
  package contract coverage atom
  waiver summary atom

home-deployment
  host readiness atom
  provider gate atom
  destructive approval state atom

joern-effect
  template registry atom
  query evidence atom
  generated schema coverage atom
```

Atoms remain read-side. They do not write EventLog events, mutate durable state,
perform provider actions, call external services, or own scheduler/resource
lifecycle. Effect services own actions. Reactivity announces changed keys.
Atoms describe semantic views over those facts.

Rationale: semantic coverage should be observed from the application's own
reasoning graph, not from hand-maintained labels or raw line percentages. If a
package operation is meaningful, the package view graph should move coherently.

Alternative considered: keep semantic coverage as a manually authored label
catalog. Rejected because it adds too much structure and relies on agents to
guess all important semantics up front.

Alternative considered: use V8/Istanbul line coverage as the semantic gate.
Rejected because implementation coverage cannot tell whether the package's
read/reasoning view changed coherently.

### Decision: Package contracts are authored as Effect Schema values

The package contract source of truth is Effect Schema. A package contract module
authors the package boundary through Schema-backed declarations, and every
consumer decodes or encodes through the contract schema:

```txt
PackageContractSchema
  -> PackageContract
  -> derived ledger emission
  -> architecture policy decoding
  -> Nx graph metadata
  -> property evidence expectations
```

Plain TypeScript object typing is not a separate contract model. Generated
helpers may make authoring pleasant, but the authoritative validation and ledger
shape comes from Effect Schema.

Rationale: Attune is Effect-first. Using Effect Schema as the source of truth
keeps package contracts aligned with operation schemas, arbitrary generation,
runtime decoding, and evidence emission.

Alternative considered: plain TypeScript values with `satisfies`. Rejected
because it is too weak for runtime decoding and external ledger emission.

Alternative considered: a hybrid TS-first value plus Schema for export. Rejected
because it creates two conceptual contract surfaces for agents to learn.

### Decision: Package contracts act as a type-level compiler

`src/attune.package.ts` is allowed to be one of the heaviest typechecking
boundaries in the repo. The authored surface should use generated helpers rather
than raw object literals:

```ts
export const PackageContract = definePackageContract({
  packageId: "home-deployment",
  packageKind: "day0-resource-runbook",
```

The builder layer preserves literal operation ids and view ids, narrows
operation kinds, and exposes type-level helpers such as:


The implementation must make those helpers concrete enough that package
contract typechecking is a real compile-time phase, not a runtime checker with
TypeScript wrappers. The core API should be shaped around generated generic
types like:

```ts
type AttuneTypeError<Message extends readonly unknown[]> = never & {
  readonly __attuneTypeError__: Message
}

type PackageContractTypes<C extends AttunePackageContract> = {
  readonly packageId: PackageIdOf<C>
  readonly operationIds: OperationIds<C>
  readonly operations: OperationMapOf<C>
  readonly views: PackageViewsOf<C>
  readonly laws: InferredLawMapOf<C>
  readonly rpc: FuzzRpcGroupOf<C>
  readonly handlers: FuzzHandlersOf<C>
  readonly evidence: EvidenceEnvelopeOf<C>
  readonly replay: ReplayEnvelopeOf<C>
  readonly counterexamples: CounterexampleEnvelopeOf<C>
  readonly requiredServices: RequiredServicesOf<C>
  readonly typeGuidance: TypeGuidanceOf<C>
}

type AssertPackageContract<C extends AttunePackageContract> =
  AssertUniqueOperationIds<C> &
  AssertKnownOperationKinds<C> &
  AssertOperationMetadata<C> &
  AssertTouchedViewsExist<C> &
  AssertLawCompatibility<C> &
  AssertRpcCompatibility<C> &
  AssertLayerRequirements<C>
```

Each generator should also emit a compile-only assertion module, for example
`src/attune.package.typecheck.ts`, that imports the authored contract and forces
TypeScript to evaluate the important derived types:

```ts
import { PackageContract, PackageLayer, PackageTestLayer } from "./attune.package"

type Contract = typeof PackageContract

type _contract = AssertPackageContract<Contract>
type _handlers = AssertExactHandlers<Contract, typeof PackageFuzzHandlers>
type _properties = AssertPropertyHarnesses<Contract, typeof PackageProperties>
type _packageLayer = AssertLayerProvidesPackageServices<Contract, typeof PackageLayer>
type _testLayer = AssertLayerSatisfiesRequiredServices<Contract, typeof PackageTestLayer>
type _typeGuidance = AssertTypeGuidanceComplete<Contract, typeof PackageTypeGuidance>
```

These modules exist to make type failures easy for Nx, editors, and agents to
surface. They do not own runtime behavior and should not introduce runtime
side effects.

The type system should model the package contract as a set of correlated maps,
not as unrelated string fields:

```txt
operation tuple
  -> duplicate-id check
  -> OperationIds<C>
  -> OperationMapOf<C>

operation kind
  -> required metadata type
  -> allowed law type
  -> inferred law type
  -> required evidence fields
  -> required type-guidance partitions

Effect Schema input/output/error
  -> decoded type
  -> encoded replay type
  -> RPC payload/success/error type
  -> FastCheck arbitrary type
  -> type-guidance partitions

declared package views
  -> known Reactivity keys and atom ids
  -> allowed `touches(...)` references
  -> inferred view movement laws
  -> atom graph coverage obligations

Effect Layer requirements
  -> package-owned service outputs
  -> required external services
  -> PackageTestLayer satisfaction proof
```

Kind-specific operation builders should be typed as metadata gates. For
example:

```ts
type OperationMetadataFor<K extends OperationKind> =
  K extends "resource-provider" ? ResourceProviderMetadata :
  K extends "projection" ? ProjectionMetadata :
  K extends "generator" ? GeneratorMetadata :
  K extends "policy-rule" ? PolicyRuleMetadata :
  K extends "joern-template" ? JoernTemplateMetadata :
  K extends "atom-family" ? AtomFamilyMetadata :
  K extends "event-facade" ? EventFacadeMetadata :
  K extends "codec" | "query" | "command" ? BasicOperationMetadata :
  AttuneTypeError<["Unknown operation kind", K]>

type AllowedLawsFor<
  K extends OperationKind,
  Metadata,
> = SharedSchemaLaws | KindLawFamily<K, Metadata> | ViewLawsFrom<Metadata>

type InferredLawsFor<
  K extends OperationKind,
  Metadata,
> = InferSchemaLaws<Metadata> | InferKindLaws<K, Metadata> | InferViewLaws<Metadata>
```

The resulting developer experience should make invalid states unrepresentable:
a resource-provider operation cannot typecheck without observation metadata; a
destructive operation cannot typecheck without current proof and approval
schemas; a generator cannot typecheck without option/tree/output/provenance
schemas; and a handler map cannot typecheck if it is missing an operation or
contains an operation id not present in the contract.

The contract helper should reject, at typecheck time where TypeScript can
express it:

```txt
duplicate operation ids
invalid package kind or operation kind pairings
invalid law ids for an operation kind
explicit laws that duplicate or contradict inferred laws
view references to undeclared Reactivity keys or atoms
resource-provider operations missing observation metadata
destructive operations missing proof or approval schemas
projection operations missing event/state schemas
generator operations missing option/tree/output/provenance schemas
policy-rule operations missing finding schemas
operation-specific RPC handlers missing after generation
RPC payload/success/error schemas that do not match the operation contract
PackageTestLayer gaps for generated handlers, laws, or schema requirements
counterexample or replay payloads that do not match encoded input schemas
```

Type errors should be branded and agent-readable, for example:

```txt
TypeError<["Invalid law for operation kind", "resource.destructive-approval", "query"]>
TypeError<["Unknown atom id", "hostReadyAtom", "Known atoms", "hostReadinessAtom"]>
TypeError<["Missing operation handler", "nixos-anywhere-install"]>
TypeError<["Missing destructive proof schema", "nixos-anywhere-install"]>
TypeError<["PackageTestLayer missing service", "home/InstallerSshClient"]>
```

Raw `as const` contract objects may exist as generated/encoded artifacts, but
agents should normally author through typed contract builders and
`@attune/nx` generators.

Rationale: the package contract drives generated RPC, fuzz, law, atom graph,
evidence, and agent workflow surfaces. Paying a larger typechecking cost at
this boundary catches drift before Nx generates a whole family of wrong files.

Alternative considered: keep contracts as plain `as const` values with runtime
validation only. Rejected because it leaves laws, views, RPC handlers, and
evidence shapes stringly typed until generated code or runtime conformance
fails.

### Decision: Invariant ownership is type-first

The completed system assigns invariants to the strongest boundary that can
prove them. Architecture policy should become a thin repo observer around a
strong typed contract, not a large bespoke linter for facts the type system can
already know.

```txt
TypeScript contract builders
  prove local contract shape, ids, references, laws, handlers, layer needs

Effect Schema
  decodes runtime values, RPC payloads, ledgers, replay seeds, evidence

Nx / generated sync
  proves files exist, generated output is fresh, targets/docs use public API

FastCheck / runtime observation
  proves behavior, atom movement, coverage, provider state, destructive gates

attune-architecture residual policy
  checks only cross-file, filesystem, command-surface, waiver, and stale-output
  facts that cannot be made local type errors
```

TypeScript should own at least these package-local invariants:

```txt
duplicate operation ids
invalid package kinds and operation kinds
invalid law ids for an operation kind
required inferred laws missing or contradicted
undeclared Reactivity keys or atom ids
missing kind-specific schemas and metadata
destructive providers missing proof, approval, or observation schemas
projection/generator/policy/Joern operations missing their typed metadata
RPC handler maps that are missing operations or contain extras
handler payload/success/error mismatch with operation schemas
evidence, replay, and counterexample types drifting from encoded schemas
PackageTestLayer requirements not satisfied by generated harnesses
```

Effect Schema remains authoritative for values crossing runtime boundaries:
contract decoding, generated ledger emission, Effect RPC payload/success/error
values, persisted counterexamples, corpus seeds, property evidence, waiver
summaries, and coverage records.

Nx and `attune-architecture` should only check facts TypeScript cannot see:
missing `src/attune.package.ts` files, stale generated files, raw command
surfaces in `project.json` or docs, package existence, changed affected graph
edges, generated ledger drift, expired waivers, and renamed package identity.

FastCheck and provider/runtime evidence own non-static claims: idempotence by
observation, destructive operation safety, atom/Reactivity graph movement,
coverage reachability, mutation survival, scheduler/provider side effects, and
hidden host-state reads that bypass typed configuration boundaries.

Rationale: this is the main simplification lever. The repo should spend type
complexity once at the package boundary so agents get immediate, local,
repairable failures instead of learning a large after-the-fact policy system.

### Decision: Law inference is metadata-driven and type-constrained

Operation authors should not list every base law by hand. The preferred form is
`laws: inferLaws()` with optional typed custom-law extensions and explicit
waivers for unauditable cases.

The inferred law set is computed from several typed inputs:

```txt
operation kind
  selects the base law family

input/output/error schemas
  infer decode, output, typed-error, redaction, encoded replay, and schema
  partition coverage laws

Schema equivalence and annotations
  infer semantic equality, ignored fields, redaction, examples, partitions,
  shrinking hints, and fuzz coverage labels

declared views
  infer Reactivity key movement, atom refresh, view diff, and dead-view laws

resource observation metadata
  infers observed-state idempotence and no-repeat destructive action laws

destructive metadata
  infers current proof plus current approval laws

projection event/state metadata
  infers deterministic replay, duplicate handling where declared, and state
  transition coverage laws

generator option/tree/output/provenance metadata
  infers deterministic output, checked generated files, and provenance laws

policy finding metadata
  infers deterministic findings, no duplicate diagnostics, waiver behavior, and
  final-ratchet laws

Joern/template metadata
  infers deterministic emitted CPGQL, evidence decode, and template registry
  coverage laws
```

Explicit law ids are still supported for readability and package-specific
extensions, but they are constrained by operation kind and contract metadata.
For example, a `query` operation cannot claim
`resource.destructive-approval`, and a resource-provider operation cannot claim
destructive approval unless it declares destructive proof and approval schemas.

Inferred laws also drive generated artifacts:

```txt
law descriptors
property harness registration
Effect RPC evidence fields
agent findings
coverage expectations
waiver summaries
```

Rationale: law inference is where the contract stops being a manifest and
starts being an executable architecture. The more the repo can infer from
schemas, operation kind, views, and resource metadata, the less agents have to
remember.

Alternative considered: require explicit law arrays for every operation.
Rejected because base laws would become cargo-cult strings and agents would
miss important laws when adding new operation kinds.

### Decision: Canonical services use `Effect.Service`

New generated services use the canonical Effect service class shape:

```ts
export class FooService extends Effect.Service<FooService>()("pkg/FooService", {
  accessors: true,
  effect: Effect.gen(function* () {
    return {
      operation: (input: Input) => ...
    }
  }),
}) {}
```

The generator owns the public skeleton: service id, operation declarations,
schema attachment, accessors, default/test layer exports, and package-contract
registration. Agents fill implementation inside generated boundaries.

Lower-level `Context.Tag` service definitions are allowed only when the package
contract records a waiver. Valid waiver categories include:

```txt
third-party boundary shim
resource lifecycle requiring lower-level scoped control
temporary migration adapter
generated service family
Effect beta typing limitation
cyclic dependency untangling
legacy compatibility boundary
```

Each waiver records owner, reason, expiration or review target, and validation
that still applies.

Rationale: a single visible service grammar makes property harness generation,
layer discovery, policy checks, and agent edits much simpler.

Alternative considered: standardize on `Context.Tag` everywhere. Rejected
because it keeps more boilerplate and gives generators less semantic structure
for accessors and service construction.

Alternative considered: allow either shape freely. Rejected because the whole
point is reducing agent choice at repeated boundaries.

### Decision: Effect DI owns service dependencies; Nx owns project orchestration

Effect DI is the source of truth for service dependencies and runtime layer
composition. Nx is the source of truth for project graph orchestration,
affected execution, caching, public commands, generator entrypoints, and policy
targets.

The bridge is derived:

```txt
Effect package contract
  -> package service graph
  -> package-level dependency summary
  -> Nx project graph metadata and edges
  -> inferred targets and affected runs
```

Nx must not become a duplicate DI container. It should lift package-level facts
from the contract into the project graph so agents can ask "what changed?" and
"what checks run?" using one command surface.

Rationale: Effect is already the runtime substrate. Nx is excellent at package
graph scheduling and generator workflow. Keeping the boundary clean prevents a
second application model.

Alternative considered: make every Effect service an Nx project. Rejected
because that would create too much graph noise and would make package-local
refactors too expensive.

### Decision: Nx discovers contracts and creates inferred targets

The Attune Nx plugin will discover `src/attune.package.ts` and create or verify
package targets such as:

```txt
<pkg>:sync-package-contract
<pkg>:service-conformance
<pkg>:property
<pkg>:coverage-conformance
<pkg>:check-generated
```

Workspace targets compose those package targets:

```txt
workspace:package-contracts-check
workspace:property-evidence
workspace:coverage-conformance
workspace:policy-fast
workspace:policy-proof-pressure
```

`workspace:policy-fast` should run the cheap contract and property tier.
`workspace:policy-proof-pressure` should include first-class coverage-search,
mutation, fuzz, and Joern/container-backed campaigns where applicable.
`workspace:package-contracts-check` is a focused diagnostic and repair target
for package-contract, law, generator-provenance, typed-surface, and waiver
issues. It is not a third policy lane. The completed public surface
intentionally does not define `workspace:policy-architecture`; architecture
checks either compose into `workspace:policy-fast`,
`workspace:policy-proof-pressure`, or focused diagnostic targets.

Rationale: agents should run Nx targets, not remember private scripts. Inferred
targets let the contract shape add checks consistently across packages.

Alternative considered: manually add targets to every `project.json`. Rejected
as drift-prone. Project-local explicit targets are still allowed when they
customize heavier tiers.

### Decision: A small generic Nx executor family owns repeated workflows

The implementation should avoid one executor per tool or package. The public
surface is still Nx-owned, but repeated behavior is routed through a compact
executor family:

```txt
attune:package-check
  contract, service conformance, atom graph conformance, property evidence,
  coverage conformance, typecheck/test/lint composition for a package

attune:generated
  sync generators, generated ledger emission, stale generated output checks,
  generator provenance verification

attune:toolchain
  typed invocation of real external toolchains such as Nix, Joern, Arion,
  Alchemy, Vite, Kubernetes generators, mutation runners, and workerized fuzz
  campaigns
```

Specialized executors may exist only when a generic executor would hide
important typed options or evidence. The default should be to add an operation
kind, contract metadata, or executor option before adding a new executor class.

Rationale: Nx should own the public surface without replacing shell-command
sprawl with executor sprawl. A small executor family keeps options typed while
leaving behavior derivable from the package contract.

### Decision: Nx replaces package scripts and wrapper command surfaces

The completed migration does not keep the old command surface around as a
wrapped implementation detail. Agents and humans interact with the repo through
Nx targets and Nx generators:

```txt
nx run workspace:policy-fast
nx run workspace:policy-proof-pressure
nx run workspace:package-contracts-check
nx run <project>:typecheck
nx run <project>:test
nx generate @attune/nx:<generator>
```

Package-local `scripts`, root or package codex wrapper scripts, raw
package-manager commands, raw `nix develop --command ...` command strings, and
ad hoc shell fragments are migration debt. They must be removed as packages
migrate. Where a package needs a real toolchain action, such as Joern schema
extraction, Nix/container fuzzing, Vite serving, CRD generation, or
Alchemy/provider execution, that behavior is owned by a typed Nx executor or
inferred target whose options are declared in package contract metadata.

Nix remains the workspace toolchain substrate where needed, and package
managers remain installation machinery, but neither is exposed as the package
workflow grammar. The public rule is:

```txt
outside implementation internals:
  Nx command or Nx generator

inside typed executor implementation:
  toolchain-specific process execution may exist
  contract metadata records inputs, outputs, env/config needs, resource tier,
  worker budget, destructive/resource-provider gates, and replay evidence
```

The final ratchet rejects package-local `scripts`, arbitrary `nx:run-commands`
shell strings, and project command strings that invoke `pnpm`, `corepack`,
`npm`, `yarn`, `nix`, `bash`, `sh`, `tsx`, `tsc`, `vitest`, `stryker`,
`arion`, `alchemy`, or workspace wrapper scripts directly. Temporary waivers
may exist only before the final ratchet; completed packages must use typed Nx
executors or inferred targets.

Rationale: Nx owning the public surface only matters if the old surfaces stop
being places agents can copy from. A typed executor keeps heavy toolchain facts
computable while removing the package-by-package shell vocabulary that caused
drift.

Alternative considered: let project targets keep arbitrary `run-commands`
strings and document only `nx run`. Rejected because it preserves the hidden
public API as untyped shell.

Alternative considered: remove Nix/toolchain use entirely from targets.
Rejected because proof-pressure, container, Joern, Alchemy, and platform
targets still need real toolchains; the change is that Nx owns the typed
interface to them.

### Decision: Framework materialization replaces hand-maintained report truth

Source BOM, generator shape, waiver, coverage, architecture summaries, and
protocol reports are not final semantic workflow surfaces. The framework
materializes protocol facts into local runtime/cache and generated source
required by build/typecheck. Source BOM and generator-shape manifests may
remain as legacy migration scaffolding or temporary compatibility views, but
the final framework posture is diagnostics plus deterministic source
generation, not checked-in report artifacts.

```txt
primary truth:
  - Nx project graph
  - package contract modules
  - public framework DSL declarations
  - generated source required by build/typecheck
  - local gitignored Protocol Runtime cache
  - generator invocation/provenance metadata
  - checked generated files
  - Effect DI layers
  - Reactivity keys and atom package view graphs
  - FastCheck/evidence producer observations in local runtime/cache

allowed projections:
  - TypeScript language-service diagnostics, quick info, actions, code lenses
  - Nx CLI/check output
  - optional gitignored debug reports
  - optional CI artifacts that are not committed

legacy migration scaffolding:
  - Source BOM shards
  - generator shape manifests
  - temporary waiver/reconciliation summaries
```

Agents update source inputs through framework DSL edits and deterministic Nx
generators. Policy should reject checked-in ProtocolDelta reports, obligation
reports, evidence summaries, Markdown/JSON architecture summaries, and manual
derived report truth.

Rationale: BOM-like files are valuable because they are diffable review
surfaces during migration. They become dangerous when humans or agents treat
them as the workflow and duplicate facts that the framework can compute from
source declarations, runtime/cache state, generator provenance, Nx, and atom
graphs.

Alternative considered: keep a manually curated BOM alongside generated
contract metadata. Rejected because it leaves agents with two truths to update
and makes generator provenance drift harder to diagnose.

### Decision: FastCheck becomes the generated boundary auditor

Every package operation with Effect Schema metadata gets a generated property
harness. The harness:

```txt
constructs the package test layer
generates inputs from Effect Schema
executes the public Effect service accessor or operation
validates success outputs against output schemas
validates typed failures against error schemas
checks inferred law descriptors from the operation kind and metadata
checks package-specific law extensions when declared
records atom/Reactivity graph movement and transition coverage
records seed, path, shrink, and counterexample metadata
```

Operation kinds define law-inference families:

```txt
codec
query
command
projection
event-facade
atom-family
resource-provider
generator
policy-rule
joern-template
```

This full taxonomy is in scope for the heavy migration. Packages may start with
minimal laws for a kind, but operations must be classified into the canonical
taxonomy rather than inventing package-local kind names.

Examples:

- A `codec` operation checks decode/encode round trips where possible.
- A `projection` operation checks deterministic replay and duplicate-handling
  laws.
- A `resource-provider` operation checks observed-state idempotence and blocks
  destructive transitions without current proof.
- A `generator` operation checks deterministic output from the same tree and
  options.

The implementation uses a compact shared law kernel first:

```txt
schema law
  generated inputs, outputs, and typed failures decode through Effect Schema

determinism/idempotence law
  repeated observation or replay yields stable evidence where the operation
  declares stability

side-effect boundary law
  pure/query/codec/atom operations do not perform undeclared writes or hidden
  configuration reads

view-movement law
  operations that declare meaningful state change move the declared
  Reactivity/atom package view graph
```

Canonical operation kinds map to those primitives and may add package-specific
laws where the domain needs them. Resource providers, generators, projections,
and Joern templates still get stronger default law mappings, but the migration
does not require a bespoke law framework for every operation kind before the
first package can conform.

Rationale: agents should not need to write a bespoke test before new code gets
meaningful pressure. The generated harness gives every package a minimum audit
surface immediately.

Alternative considered: require each package to write property tests manually.
Rejected because it recreates uneven coverage and makes the uniform contract
less useful.

### Decision: Effect RPC is an optional generated package harness protocol

The long-term generated fuzzer harness may talk to packages through generated
internal Effect RPC groups when that backend is compatible. A package contract
can generate:

```txt
PackageFuzzRpc
  control RPCs:
    reset
    snapshot
    observe
    flushEvidence
    replayCounterexample
    getCoverage
    getAtomGraph

  operation RPCs:
    invoke<OperationName>
```

Each generated RPC uses Effect Schema for payload, success, and error values.
The generated handler layer is provided by `PackageTestLayer` and calls only
the public service accessor, generator, provider, policy, projection, or query
boundary declared in the package contract. The fuzzer core becomes an RPC
client plus law/evidence interpreter; it does not import package-private
implementation functions or invent package-specific invocation envelopes.

The authored source of truth remains the Attune Protocol declaration exposed
through `src/attune.package.ts`; the portable form is the Effect Schema
Protocol Descriptor and the evidence-producing state lives in the Protocol
Runtime. Effect RPC is a generated internal harness protocol that may be
derived from that contract when the workspace dependency stack supports it:

```txt
src/attune.package.ts
  -> OperationContract schemas
  -> generated PackageFuzzRpc
  -> generated PackageFuzzRpcLive handlers
  -> generated property/fuzz runner
  -> Schema-coded evidence and findings
```

The default generated style is one RPC per public auditable operation so
payload, success, and error schemas stay precise. The fuzzer may also generate
a uniform registry from those RPC descriptors so it can iterate over operations
abstractly by package id, operation id, seed range, law ids, and coverage
expectations.

RPC values crossing worker, process, corpus, replay, and report boundaries are
Schema-coded:

```txt
operation payloads
operation successes and typed errors
atom/Reactivity observations
coverage observations
law results
counterexamples
replay seeds
agent findings
```

Rationale: Effect RPC is the native version of the schema invocation envelope.
It can give the fuzzer a typed protocol, handler layers, Schema payloads, and
client/server separation without creating a custom Attune RPC format. It is not
the root primitive, and runtime RPC is not required while the current Effect 4
beta workspace and available `@effect/rpc` package remain compatibility-risky.
The root primitive is the Schema-coded protocol descriptor plus generated
obligations, evidence, and runtime deltas.

Alternative considered: define custom `InvocationEnvelope` and `ExitEnvelope`
schemas and dispatch them manually. Rejected because Effect RPC already
provides the Schema-backed protocol and Layer-backed handler model.

Alternative considered: have the fuzzer import package functions directly.
Rejected because it bypasses the package boundary and makes private
implementation detail part of the test surface.

### Decision: Effect Schema arbitraries are the input source

Operation inputs are generated from Effect Schema using Effect's Schema to
Arbitrary integration, such as `Arbitrary.make(schema)`, which returns a
`fast-check` `Arbitrary` for values accepted by the schema. Package harnesses may
compose or weight those arbitraries for coverage search, but the base input
space remains the operation's Effect Schema.

Generated property harnesses must decode generated inputs, successful outputs,
and typed errors through the package contract schemas. Custom FastCheck
arbitraries are allowed only as refinements around the Schema-derived arbitrary
or as explicit waivered escape hatches for schemas that cannot currently produce
efficient values.

Rationale: the package contract is Effect-first. If tests generate from a
different source than the contract schema, agents can accidentally test a world
the operation does not declare.

Alternative considered: author custom FastCheck arbitraries for every
operation. Rejected because it adds a second input grammar and pushes too much
manual test design back onto agents.

### Decision: Coverage search uses transforms before filters

Generated property harnesses start with the operation input schema:

```txt
Effect Schema input contract
  -> Arbitrary.make(schema)
  -> schema-valid base arbitrary
  -> coverage-search transforms and weighting
  -> corpus replay and targeted seeds
  -> measured filters only when unavoidable
```

Effect Schema transforms describe contract-level decoding and encoding behavior.
FastCheck `map`, `chain`, `oneof`, and weighted composition describe search
behavior over values already rooted in the schema. Search transforms may bias
the generator toward missing schema variants, operation states, atom graph
movement, V8/Istanbul coverage points, or persisted counterexample corpus
entries, but they must not become a second package input contract.

Filters are allowed only as a last-mile search constraint when the predicate is
cheap and the acceptance rate stays high. The property runner records filter
identity, rejection counts, acceptance rate, and whether a filter came from a
schema refinement, operation precondition, corpus replay, or temporary harness
workaround. High rejection rates are reported as generator-quality findings so
agents can tighten the Effect Schema, add a constructive search transform, or
promote a corpus entry instead of hiding weak generation behind `.filter`.

Rationale: coverage-guided search needs to steer generation, but the repo
should still have one declared input grammar. Transforms let FastCheck explore
useful neighborhoods without asking agents to author a parallel arbitrary
catalog. Measured filters keep impossible or irrelevant cases from dominating
the run.

Alternative considered: let every harness author custom arbitraries directly.
Rejected because it recreates hand-maintained test grammar and weakens contract
coverage.

Alternative considered: ban FastCheck filters entirely. Rejected because some
operation preconditions are cheaper to express as measured predicates during a
search run, especially while the schema is being refined.

### Decision: Type guidance is a first-class fuzzer signal

TypeScript types are erased at runtime, so the fuzzer cannot inspect the
compiler's internal types directly while it runs. The implementation must make
types useful to the fuzzer by reifying contract-derived type facts into
Schema-backed, generated guidance that is still checked by TypeScript.

Each package contract should generate a `PackageTypeGuidance` artifact:

```ts
export const PackageTypeGuidance = defineTypeGuidance(PackageContract, {
  operations: {
    "nixos-anywhere-install": {
      inputPartitions: [
        { id: "disk-proof.present", from: "schema.required-field" },
        { id: "disk-proof.missing", from: "schema.negative-case" },
        { id: "approval.current", from: "destructive.approval-schema" },
      ],
      outputPartitions: [
        { id: "install.observed", from: "output.discriminant" },
        { id: "install.applied", from: "output.discriminant" },
      ],
      errorPartitions: [
        { id: "ssh.unreachable", from: "typed-error.discriminant" },
      ],
      lawPartitions: [
        { id: "resource.observed-idempotence", from: "inferred-law" },
        { id: "resource.destructive-approval", from: "inferred-law" },
      ],
      viewPartitions: [
        { id: "host-readiness.moves", from: "touches.reactivity-key" },
      ],
    },
  },
})

type _typeGuidance =
  AssertTypeGuidanceComplete<typeof PackageContract, typeof PackageTypeGuidance>
```

`PackageTypeGuidance` is not a hand-authored fuzzer grammar. It is generated or
sync-generated from the package contract, Effect Schema AST, Schema annotations,
operation kind, inferred laws, typed error schemas, declared views, and
operation metadata. Agents may add package-specific guidance only through typed
Schema annotations, law extensions, or contract metadata, not through an
untyped side file.

The type-guidance model should include at least:

```txt
operation id partitions
operation kind partitions
input schema union/discriminant variants
literal, enum, template-literal, brand, and refinement partitions
optional/required field presence
collection empty/single/many boundary cases
numeric/string/date boundary cases declared by Schema annotations
encoded-vs-decoded replay partitions
output discriminants
typed error discriminants and expected error paths
inferred law ids
custom law ids
declared Reactivity key and atom movement obligations
resource/destructive observation states
projection event/state transitions
generator option/tree/provenance cases
policy finding severity/category cases
Joern template registry/query-shape cases
```

The fuzzer scores generated cases across a novelty vector:

```txt
type-guidance partition novelty
Schema branch novelty
atom/Reactivity graph movement novelty
operation transition novelty
law-observation novelty
V8/Istanbul implementation coverage novelty
mutation/weak-oracle pressure
counterexample/corpus replay value
```

Search transforms may then bias toward missing type partitions exactly the same
way they bias toward missing atom graph movement or V8 coverage. For example,
if the type guidance says an operation has a typed error discriminant that has
never appeared, the next targeted run may increase weight for inputs that can
reach that error path. If a branded or refined schema partition is always
filtered out, the finding should be a generator-quality or schema-guidance
defect for agents.

Rationale: this is how the fuzzer becomes type-guided without pretending that
runtime JavaScript can read TypeScript types. Types generate and validate the
guidance; Effect Schema reifies it; FastCheck uses it for generation and
coverage search.

### Decision: `@fast-check/worker` powers workerized property execution

Parallel and long-running property targets use `@fast-check/worker` for
dedicated worker execution of generated predicates. The generated property
modules are structured so properties can be hoisted at module scope and wrapped
with `propertyFor(new URL(import.meta.url))`, while tests and Nx runners call
the worker-aware `assert` exported by `@fast-check/worker`.

The worker runner supports three execution tiers:

```txt
commit tier:
  Nx schedules affected packages.
  Worker usage is optional and defaults to shared file-level workers where
  package risk or sync-loop protection justifies it.

push/proof-pressure tier:
  Nx shards packages and operations.
  @fast-check/worker runs properties in dedicated workers with per-property
  timeouts and bounded worker counts.

fuzz/coverage-search tier:
  Nx shards by package, operation, seed range, and coverage corpus.
  @fast-check/worker uses property-level or predicate-level isolation for
  risky operations, infinite-loop protection, and long-running campaigns.
```

The default `randomSource` is `main-thread` so generated values remain
serializable and shrinking remains available. Targets may opt into
`randomSource: "worker"` only when generated values are not serializable across
workers; those runs must record that shrinking is limited or unavailable.

`isolationLevel` is selected by target tier:

```txt
file       -> shared worker reuse for cheap packages
property   -> one worker pool per property for heavier operations
predicate  -> one worker per predicate run for high-risk or hang-prone inputs
```

Rationale: fast-check is excellent at generation, replay, and shrinking, but a
synchronous infinite loop can block the JavaScript main thread and prevent
reporting or shrinking. `@fast-check/worker` gives package audits isolation,
timeouts, and worker execution while keeping the same FastCheck property model.

Alternative considered: rely only on Nx parallelism. Rejected because Nx can
parallelize packages, but it cannot isolate a single hanging predicate inside a
test process.

Alternative considered: build a custom worker pool. Rejected because
`@fast-check/worker` is the fast-check ecosystem package designed for running
predicates in dedicated workers.

### Decision: Atom/Reactivity graph coverage is the primary semantic gate

Atom/Reactivity graph coverage is the contract-level semantic signal:

```txt
service operations exercised
schema variants exercised
Reactivity keys mutated
base atoms refreshed
derived atoms recomputed
package view atoms changed
state transitions reflected in atom views
operation-kind laws checked
expected error paths hit
counterexamples replayed
```

V8/Istanbul coverage is also first-class input to the search loop. It is not the
primary pass/fail contract, but it helps detect dead harnesses, unexplored
branches, and generated inputs that fail to reach meaningful implementation
paths. Coverage-guided property targets may use V8/Istanbul deltas together with
missing atom/Reactivity graph movement and transitions to bias subsequent
FastCheck runs.

The implementation model is batched and deterministic:

```txt
commit tier:
  run bounded cases
  collect package-level coverage around the batch
  retain only useful seeds/corpus entries

proof-pressure tier:
  shard by package, operation, seed range, and corpus partition
  collect per-worker or per-shard coverage
  merge evidence by stable package/operation/coverage identities

fuzz/search tier:
  loop over missing semantic partitions and new V8 coverage points
  replay retained seeds first
  bias Schema-derived arbitraries with transforms and weighted corpus entries
  report weak-oracle and dead-harness findings for agents
```

Useful coverage deltas retain the seed, shrink path when available, generated
value summary, transform/filter metadata, worker/shard metadata, and the
coverage point reached. The merge step de-duplicates by package id, operation
id, source coverage point, atom graph edge, seed, shrink path, worker id, and
shard id. If implementation code is covered but no expected atom graph movement
or law observation occurs, the finding is a likely weak oracle or missing
invariant. If generated cases run but operation implementation coverage is
absent or implausibly shallow, the finding is a dead or miswired harness.

The conformance gate remains based on package view graph coverage expectations.
The search loop uses code coverage as feedback for case generation and
diagnostics, especially dead-harness detection.

Rationale: line coverage is useful but weak for architecture. A package
boundary knows that "host readiness changed" or "decision packet refreshed"
matters because the package view graph moved, not because a specific line ran.

Alternative considered: use only Vitest/V8 coverage. Rejected because it does
not understand service contracts, operation laws, schema partitions, Reactivity
keys, or atom graph movement.

Alternative considered: keep V8/Istanbul diagnostics-only until the semantic
label model proves itself. Rejected because coverage-guided search is important
enough to design into the first heavy migration; fast local enforcement can
remain bounded while proof-pressure/fuzz targets run the stronger loop.

### Decision: Configuration and environment reads become DI boundaries

Package contracts must identify configuration dependencies as Effect services or
config-backed layers. Random direct environment reads are treated as hidden
dependencies and fail strict service conformance unless explicitly waived.

Rationale: generated property audits need deterministic test layers. Hidden
environment reads make agents and CI disagree about package behavior.

### Decision: Waivers are local source with generated review summaries

Waivers live in `src/attune.package.ts` as Schema-backed contract data next to
the package boundary they justify. A root waiver summary is generated from those
local sources for review ergonomics:

```txt
src/attune.package.ts
  -> attune.waivers.generated.json
  -> attune.waivers.generated.md
```

The generated summary is never the source of truth. Policy rejects stale or
manually drifted summaries.

Rationale: waivers are architectural exceptions. They must stay close to the
code that needs the exception, while humans and agents still need a repo-wide
view of what the migration is tolerating.

### Decision: Agent-facing guidance becomes diagnostics-first

The new agent interface is:

```txt
Read language-service diagnostics and Nx check output.
Open the referenced src/attune.package.ts declaration.
Use an @attune/nx generator or sync generator.
Implement inside generated Effect.Service boundaries.
Update Schema-backed operation metadata and laws.
Expose or update package Reactivity keys and atoms.
Run Nx-owned conformance, property, and coverage targets.
Report validation results and remaining diagnostics.
```

`AGENTS.md`, cloud-environment docs, generator docs/templates, and OpenSpec
implementation guidance must all explain the same interface. Agents should not
be told to hand-author repeated service shapes, ad hoc package scripts, raw
manifest entries, checked-in report files, raw SQLite rows, raw ProtocolDelta
files, or package-local testing idioms when a framework diagnostic, generator,
or Nx target owns that surface.

Rationale: this spec changes how agents interact with the repository. The
instructions need to be as uniform as the code shape, or the next batch of
agents will recreate the drift this change is designed to remove.

### Decision: `attune-architecture` is the final policy package

The final package name is `attune-architecture`, with final package path
`packages/attune-architecture`. The old architecture-lint identity is only a
historical pre-rename migration fact and must not survive as the final package
id, binary, documentation name, generated ledger owner, or public package
surface.

`attune-architecture` owns contract decoding, policy rules, Nx graph facts,
generated ledger validation, type-inference diagnostics, waiver summaries,
command-surface checks, property/evidence conformance, and the final ratchet
reports. It does not expose a separate `workspace:policy-architecture`
umbrella target; architecture policy composes into the minimal Nx surface
defined above.

### Decision: Migration is package-by-package but contract-gated

The initial implementation can migrate packages in phases, but the target state
is every active package conforming:

```txt
attune-nx
attune-architecture
effect-oxlint-policy
attuned-discovery
cocoindex-effect
joern-effect
joern-effect-properties
attune-foldkit
attune-pi-agent
platform-alchemy-k8s
home-deployment
```

The migration should be generated by package archetype wherever possible:

```txt
generator/tooling packages
  attune-nx, attune-pi-agent generator surfaces

policy packages
  attune-architecture, effect-oxlint-policy

product/runtime packages
  attuned-discovery, cocoindex-effect, attune-foldkit, attune-pi-agent

proof packages
  joern-effect, joern-effect-properties

platform/resource packages
  platform-alchemy-k8s, home-deployment
```

Packages may start with a minimal contract that records existing services and
waivers, then graduate to generated `Effect.Service` contracts and generated
property harnesses. Policy can ratchet from diagnostics-only to required as
packages migrate.

The completed implementation must not leave migration-only material behind.
Temporary adapters, aliases, diagnostics-only exceptions, manual BOM source
fields, checked-in protocol reports, unowned waiver categories, compatibility
exports, and transitional generated files must be removed before final ratchet.
Duplicate public command surfaces, stale package scripts, arbitrary shell
targets, and manually maintained derived ledger/report truth also have no
final-ratchet waiver path. Long-lived waivers remain
available only for genuine non-migration architecture exceptions, such as a
resource boundary that must keep a lower-level Effect service shape, and those
waivers require owners and review dates.

Rationale: this is a repo-wide contract, but forcing every package to become
perfect in one implementation pass would create too much churn.

### Decision: Implementation runs through parallel subagent waves

The migration should be executed as an aggressive in-place fork using parallel
subagent waves. `agent-migration-plan.md` is the operational plan for
implementation agents, validation agents, ownership boundaries, handoff
packets, integration cadence, and wave exit criteria.

Subagents are expected to do real implementation and validation work. Explorer
agents may answer narrow questions, but the default delegation unit is a
worker with a disjoint write scope and a concrete patch. Validation agents add
or tighten tests, run Nx/OpenSpec checks, and return actionable findings
against the owning implementation slice.

The waves are:

```txt
Phase 0: foundation freeze and worktree survey
Phase 1: contract type kernel
Phase 2: Nx generator and executor surface
Phase 3: property, RPC, worker, and evidence runtime
Phase 4: tooling package migration
Phase 5: product package migration
Phase 6: proof package migration
Phase 7: platform and Day-0 resource migration
Phase 8: docs, agent contract, and final ratchet
```

Each phase must finish with a handoff packet and validation evidence. The final
ratchet is not complete until every active package is migrated and the old
surfaces are removed.

## Risks / Trade-offs

- [Risk] The package contract becomes another hand-maintained manifest. ->
  Mitigation: generate it with `@attune/nx` and sync generators; make derived
  ledgers check-only outputs.
- [Risk] Generated property audits produce shallow confidence if schemas are
  too broad or laws are missing. -> Mitigation: require operation kinds,
  semantic labels, and package-specific laws for promoted services.
- [Risk] Property runs become too slow for every commit. -> Mitigation: define
  tiers and run cheap deterministic package audits in `policy-fast`, with
  first-class proof-pressure/fuzz campaigns available through separate Nx-owned
  targets.
- [Risk] Canonical `Effect.Service` migration disrupts existing lower-level
  resource code. -> Mitigation: support explicit waivers and migrate resource
  providers through adapters first.
- [Risk] Nx graph metadata and Effect DI graph facts drift. -> Mitigation:
  derive Nx metadata from contract discovery and fail `nx sync:check` or the
  Attune sync target when generated artifacts are stale.
- [Risk] Agents overfit to satisfying generated laws while missing domain
  behavior. -> Mitigation: package-specific law packs remain first-class and are
  promoted from counterexamples and review findings.

## Migration Plan

1. Extend `@attune/nx:effect-service` to generate the canonical
   `Effect.Service` shape, operation schema slots, test layer, and contract
   registration.
2. Add `@attune/nx:package-contract` or a sync generator that creates
   `src/attune.package.ts` for an existing package.
3. Add `@attune/nx` generators or sync generators for package view atoms,
   Reactivity keys, and atom graph registration.
4. Add contract discovery to the Attune architecture package and expose a
   workspace check that reports missing contracts, services, schemas, layers,
   and waivers.
5. Add an Nx project graph plugin or sync target that derives package-level DI
   metadata, project tags, dependency summaries, and inferred targets from
   package contracts.
6. Extend the property infrastructure so FastCheck harnesses can be generated
   from operation schemas and operation-kind law packs.
7. Add property evidence emission and an atom/Reactivity graph coverage summary
   format.
8. Wire cheap service/property conformance into `workspace:policy-fast` and
   deeper campaigns into `workspace:policy-proof-pressure`.
9. Migrate packages in rings:
  - tooling packages: `attune-nx`, `attune-architecture`,
     `effect-oxlint-policy`
   - core product packages: `attuned-discovery`, `cocoindex-effect`,
     `attune-foldkit`, `attune-pi-agent`
   - proof packages: `joern-effect`, `joern-effect-properties`
   - platform packages: `platform-alchemy-k8s`, `home-deployment`
10. Ratchet policy from diagnostics-only to required once every active package
    has a contract and at least a minimal generated property harness.
11. Replace manual BOM/shape edits with framework diagnostics, local cache, and
    sync/check targets that validate source declarations, generated source,
    generator provenance, Nx, and atom graphs.
12. Remove migration-only aliases, manual source ledgers, duplicate public
    script surfaces, and expired waivers before the ratchet is considered
    complete.
13. Execute the migration using `agent-migration-plan.md`: parallel
    implementation agents and validation agents per phase, with disjoint
    ownership, concrete handoff packets, and integrated validation before
    moving to the next wave.

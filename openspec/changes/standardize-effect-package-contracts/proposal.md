## Why

Attune is becoming a framework for agent-legible packages, not a pile of
loosely connected tools. The root `framework/` layer defines the programming
model: package authors declare operations, schemas, services, views, laws, and
waivers; the framework derives obligations, hides runtime materialization, and
surfaces violations through TypeScript language-service diagnostics, quick
info, code actions, code lenses, and Nx checks.

Effect remains the runtime/type/schema substrate. Nx remains the deterministic
action and materialization layer. FastCheck remains the adversarial evidence
engine. SQLite/Drizzle remains the local private runtime/cache for protocol
facts. The normal developer loop should be: write framework DSL, see editor
diagnostics, apply deterministic code action, run Nx check, record evidence,
watch the diagnostic disappear.

## What Changes

- **BREAKING**: Active packages must expose a canonical Attune package contract
  instead of ad hoc service exports, hidden environment reads, private command
  scripts, or untracked boundary metadata.
- **BREAKING**: New package services must prefer the canonical
  `Effect.Service` shape generated by `@attune/nx`; lower-level `Context.Tag`
  service definitions require an explicit exception.
- Add package-level contract modules that describe services, operations,
  schemas, laws, layers, generated provenance, Reactivity keys, atom view
  graphs, and coverage expectations.
- Introduce Attune Framework as the root `framework/` layer outside
  `packages/`. `framework/` defines the programming model; `packages/` consume
  it.
- Expose Attune Protocol through `src/attune.package.ts` using public framework
  DSL helpers such as `defineAttunePackage`, operation builders, view builders,
  law helpers, and evidence producer helpers.
- Add Schema-coded Attune Protocol descriptors, obligations, generated artifact
  records, evidence events, deltas, waivers, and repair actions.
- Add a local SQLite/Drizzle-backed Attune Protocol Store behind Effect
  services. The store is the local compiler/runtime database for protocol
  state, generated artifact status, evidence, deltas, and repair suggestions,
  not a product database.
- Make package contracts heavily typed authoring surfaces. `definePackageContract`,
  package view builders, and kind-specific operation builders infer operation
  ids, Schema decoded/encoded types, valid laws, valid view references, Effect
  RPC shapes, handler exhaustiveness, evidence/replay types, and
  `PackageTestLayer` requirements.
- Generate compile-only type assertion modules and concrete contract type
  surfaces so TypeScript rejects duplicate ids, stale generated handlers,
  invalid laws, invalid views, missing kind-specific metadata, incomplete type
  guidance, and layer requirement gaps before runtime checks run.
- Assign invariants to the strongest enforcement boundary: TypeScript for
  type-expressible local contract rules, Effect Schema for runtime/encoded
  values, Nx/generated sync for repo and generated-file facts, FastCheck and
  providers for observed behavior, and `attune-architecture` only for residual
  repo-wide policy.
- Treat the package contract as the unit of uniformity. It inventories public
  auditable operations and package-level semantic views, while private helpers
  remain implementation detail covered through those boundaries.
- Allow pure or data-heavy packages to expose minimal or empty package layers
  when they do not own runtime services, instead of forcing fake DI services.
- Implement operation-kind laws through a compact shared law kernel plus
  package-specific extensions, not a large bespoke law catalog for every kind.
- Infer default law sets from operation kind, Schema metadata, declared views,
  destructive/resource-provider metadata, projection/generator metadata, and
  Schema annotations; explicit custom laws remain typed extensions rather than
  stringly package-local catalogs.
- Make atom/Reactivity graph coverage the primary semantic coverage gate for
  package operations through package-level view atoms rather than atoms for
  every internal helper.
- Derive repo-wide architecture state from Effect DI contracts and the Nx
  project graph before emitting generated BOM-like ledgers or custom checked
  artifacts.
- Hide SQLite tables, Drizzle clients, descriptor hashes, obligation derivation
  internals, generated artifact manifests, evidence indexes, diagnostic
  projection internals, and protocol store tables behind framework services.
- Make the TypeScript language service the primary rich framework view:
  diagnostics, quick info, code actions, and code lenses over framework
  diagnostics. Nx CLI/check output is the secondary view.
- Remove checked-in ProtocolDelta reports, obligation reports, evidence summary
  reports, Markdown/JSON architecture summaries, Linear/GitHub summaries, and
  cloud-agent report artifacts from the core workflow. Debug/CI reports may be
  emitted ephemerally to stdout, uploaded as CI artifacts, or written under
  gitignored cache paths only.
- Downgrade MCP to a future optional adapter over framework diagnostic/query
  services, not a core package, task, exit criterion, or implementation wave.
- Finalize the broad architecture package identity as `attune-architecture`.
  Treat the old architecture-lint identity only as historical migration
  context.
- Extend Nx integration so package contracts create inferred targets for
  service conformance, property audits, coverage conformance, and generated
  ledger synchronization.
- Keep the public Nx policy surface minimal: `workspace:policy-fast` for the
  default commit gate, `workspace:policy-proof-pressure` for heavy
  proof/fuzz/mutation campaigns, and `workspace:package-contracts-check` as a
  focused diagnostic/repair target. Do not introduce a separate
  `workspace:policy-architecture` umbrella.
- Consolidate package workflows completely behind typed Nx targets and
  generators; remove package-local scripts, codex wrapper commands, raw
  `run-commands` shell surfaces, and direct package-manager/Nix/tool invocations
  from migrated package APIs unless explicitly waivered.
- Prefer a small executor family for repeated workflow shapes: package checks,
  generated artifact sync, and typed toolchain actions.
- Generate FastCheck property harnesses from Effect Schema operation contracts
  so agents can get boundary audit pressure without writing bespoke tests first.
- Generate Schema-coded internal package harnesses from package contracts so
  the fuzzer invokes public package boundaries through typed payload, success,
  error, observation, replay, and evidence values instead of custom ad hoc
  invocation envelopes. Runtime `@effect/rpc` is an optional backend for that
  harness, not the root primitive.
- Run generated property and fuzz harnesses through `@fast-check/worker` where
  parallelism, timeouts, or isolation are needed.
- Record automatic property evidence for schema branches, service operations,
  operation-kind laws, atom/Reactivity graph movement, transitions, coverage
  labels, seeds, shrink paths, and counterexamples.
- Add type-guided fuzzing through Schema-backed `PackageTypeGuidance` artifacts
  validated by TypeScript and consumed alongside atom/Reactivity coverage,
  V8/Istanbul coverage, mutation pressure, and corpus replay.
- Migrate every active package to the canonical contract shape by package
  archetype where possible, with waivers only for documented lower-level Effect
  DI or runtime-boundary cases.
- Use `package-migration-inventory.md` as the detailed per-package checklist
  for final contract shape, atom/Reactivity views, property evidence, generic
  typed Nx executor migration, and cleanup with no dangling migration material.
- Use `agent-migration-plan.md` as the phased parallel subagent execution plan,
  with implementation agents, validation agents, ownership boundaries, handoff
  packets, and exit criteria for the aggressive in-place migration.

## Capabilities

### New Capabilities

- `effect-package-contracts`: Defines the canonical Effect DI package boundary,
  package contract module, service shape, operation metadata, layer exports,
  provenance fields, and waiver model.
- `attune-protocol-runtime`: Defines Attune Protocol, protocol descriptors,
  root `framework/` runtime/store, obligation derivation, evidence/delta
  computation, language-service diagnostics, code actions, SQLite/Drizzle
  lifecycle, information hiding, and Nx materialization rules.
- `nx-derived-architecture-graph`: Defines how Nx discovers package contracts,
  derives graph metadata and targets, runs sync generators, and owns the public
  command surface.
- `atom-reactivity-package-views`: Defines package atom view graphs,
  Reactivity key ownership, graph-movement coverage, and framework diagnostic
  projections.
- `automatic-property-evidence`: Defines generated FastCheck boundary audits,
  semantic coverage expectations, evidence emission, replay metadata, and
  conformance gates.

### Modified Capabilities

- None. No mainline OpenSpec capability specs exist yet.

## Impact

- Affects every active package under `packages/**`.
- Adds planned root framework projects outside `packages/`:
  `framework/protocol`, `framework/runtime`, `framework/sqlite`,
  `framework/language-service`, `framework/nx`, and `framework/testing`, with
  package names such as `@attune/framework-protocol`,
  `@attune/framework-runtime`, `@attune/framework-sqlite`,
  `@attune/framework-language-service`, `@attune/framework-nx`, and
  `@attune/framework-testing`.
- Affects workspace discovery because `pnpm-workspace.yaml`, Nx project
  discovery, and local tooling may need to include `framework/*` projects while
  keeping them conceptually separate from product packages.
- Affects `packages/attune-nx` generators and sync generators.
- Affects package `project.json` targets and `package.json` scripts because
  migrated packages must expose workflows through typed Nx executors or inferred
  targets rather than arbitrary shell commands.
- Affects the root Nx workspace targets, the final `attune-architecture`
  package, framework materialization/cache checks, import-boundary checks,
  no-checked-in-report policy, and precommit/push policy flows.
- Affects existing Effect service definitions that currently use ad hoc
  `Context.Tag`/`Layer` exports without a package contract.
- Affects package read-model, Reactivity, and atom surfaces across every active
  package.
- Affects property-test infrastructure in `joern-effect-properties` and any
  future package-level property harness.
- Affects implementation process: the migration is expected to run through
  parallel subagent waves with separate implementation and validation agents,
  not as a serial package-by-package cleanup.
- Adds `@fast-check/worker` as the workerized execution substrate for
  parallel/high-volume property targets while continuing to derive inputs from
  Effect Schema arbitraries.
- Treats `@effect/rpc` as an optional/future generated audit harness backend
  for Schema-backed package-boundary fuzzing. It is not the root primitive while
  the workspace remains on an Effect 4 beta stack and runtime RPC compatibility
  remains unresolved.
- Intentionally increases typechecking work at package-contract boundaries;
  this cost is acceptable because the contract is the source of generated
  package architecture, fuzz, optional RPC harnesses, law, evidence, and
  language-service diagnostics.

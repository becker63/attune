## Methodology

The analysis used the JSONL session events:

- `session_meta` and `turn_context` to identify session id, original timestamp, cwd/repo, model, and context summaries.
- `event_msg` with `token_count` to read cumulative and per-turn usage.
- `response_item` with `function_call` and `function_call_output` to measure terminal/tool calls and outputs.
- `Original token count` in CLI wrapper output to estimate pre-truncation command output.
- Actual stored JSON output bytes to estimate what was delivered/logged into the conversation.

Deduplication key:

```text
session_id + original_timestamp + cwd + first_user_prompt_snippet
```

For each duplicate group, the report keeps the rollout file with the highest cumulative token count as the canonical file.

Important caveat:

`Original token count` in terminal output can be enormous even when only a capped/truncated portion was logged back into the conversation. This report distinguishes:

- Original CLI output tokens: generated by command before truncation.
- Delivered/logged output estimate: actual stored output bytes divided by four.
- Model token totals: actual `token_count` data from Codex.

This is a subtle point: the model usually receives the truncated output, but the command still generated huge output. That creates:

- Slow commands.
- Noisy log metadata.
- The temptation to rerun narrower searches after already losing time.
- Follow-on context drift because the tool output says it was enormous.

High-value fix:

Use a “search ladder”:

1. `rg -l PATTERN ROOT`
2. `rg --count PATTERN ROOT`
3. `rg -n --max-count 20 PATTERN specific/files`
4. Read targeted files with bounded `sed`.

Never broad-search:

- `~/.codex`
- `/nix/store`
- `node_modules`
- `.next`
- `.cache`
- whole repo roots containing generated artifacts
- all configs/docs for numeric patterns without `--max-count`

### Research Refresh: Polling Has Two Different Failure Modes

A later pass over `/tmp/research/top_cmd_by_write_count.tsv` sharpened the polling picture. Among 1,369 command strings with write polling data:

| Metric | Value |
| --- | ---: |
| Total write poll calls | 5,053 |
| Total write poll original tokens | 50.2M |
| Commands with at least 20 write polls | 45 |
| Share of write poll calls from those 45 commands | 35.6% |
| Share of write poll tokens from those 45 commands | 47.3% |

The high-frequency tier splits into two very different behaviors:

| Family | High-Frequency Write Calls | Write Tokens | Command Strings |
| --- | ---: | ---: | ---: |
| Nx | 817 | 920,708 | 15 |
| Buck/SearchBench | 307 | 22,572,123 | 10 |
| Nix | 237 | 41,547 | 8 |
| OpenSpec | 147 | 164 | 3 |
| JS package scripts | 122 | 48,605 | 3 |
| Git | 85 | 37,070 | 3 |
| Python checks | 30 | 8,689 | 1 |

Interpretation:

- SearchBench/Buck polling is expensive because real evaluation output is large. It needs durable metrics artifacts and log summarization.
- Nx/OpenSpec/Git/Nix polling is often expensive because the agent waits, checks, and re-checks low-output process state. It needs better status protocols and fewer broad confidence checks.
- OpenSpec is the clearest zero-yield example: `openspec validate standardize-effect-package-contracts --type change` had 80 write polls across 79 sessions with 0 original output tokens; `openspec status --change standardize-effect-package-contracts --json` had 36 write polls across 32 sessions with 0 original output tokens.
- Some Nix builds also show long wait/poll behavior with almost no output, especially Attune host/image builds. These should expose a tiny progress/status file rather than relying on repeated empty polls.

The practical fix is therefore two-track:

1. For SearchBench-style evaluations, summarize the *content* of large outputs.
2. For Attune/OpenSpec/Nx/Nix coordination, summarize the *state of waiting* so the agent does not keep polling nearly silent processes.

- `write_stdin failed: stdin is closed for this session`
- `failed to parse function arguments: missing field session_id`
- `failed to parse function arguments: missing field cmd`

One row is especially worth filtering from future scorecards: `session_id=1930000000` was an invalid process id and produced 0 tokens. It can distort any naive session-id or poll-count aggregation if the analyzer treats session ids as quantities.

High-value fix:

Long-running command management needs an agent-visible process ledger:

```text
process id
command
started at
last poll at
expected max duration
status
log path
safe next action
```

This would reduce invalid polls, make stuck processes obvious, and let future agents distinguish "still running normally" from "lost process handle."

## Recommended Operating Rules

Use these as practical agent defaults:

1. Start architecture work with an inventory, then write a checkpoint and start a fresh implementation session.
2. Search with `rg -l` or `rg --count` before `rg -n`.
3. Never run broad `rg` over generated trees, caches, dependency folders, session logs, or `/nix/store` without a cap.
4. Redirect long command output to a log file and poll compact status.
5. Prefer `git diff --name-only` and `git diff --stat` before reading full diffs.
6. Read OpenSpec instructions once, then rely on a checkpoint.
7. Use scoped validation before workspace-wide validation.
8. Turn screenshots/mockups into reusable visual digests.
9. Keep dense prompts as files, not conversation context.
10. Make subagents return structured handoffs.

## Final Diagnosis

The token problem is mostly a workflow-shape problem.

Attune and SearchBench work is genuinely complex, especially around architecture, generated contracts, policy gates, and long-running evaluations. The expensive pattern is not “the model is verbose.” It is:

```text
large architectural context
+ repeated broad tools
+ long command polling
+ repeated validation/status rituals
+ multi-agent coordination
+ no compact durable checkpoint
= massive context replay
```

The highest-leverage fix is to move state out of chat and into compact repo-owned artifacts: checkpoints, ledgers, status JSON, architecture diffs, validation summaries, and visual digests.

If GPT Spark explores only one thing, make it the Attune workspace snapshot plus scoped validation planner. That would attack the repeated orientation and repeated broad validation loops that show up throughout the Attune logs.
